List Virtualization: How to Render 100,000 Rows Without Melting the Browser

A deep dive into the UI performance technique every frontend developer eventually needs — what it is, how it works, which npm libraries implement it, and how they compare.

The Ticket That Humbles Every Developer

It usually starts innocently. A product manager asks for a simple scrollable list of transactions. You've done this a hundred times. You fetch the data, map it into <div>s, deploy it, and move on to the next task. Clean code. Done.

Then QA opens it with a real account.

Fifty thousand rows. The tab freezes for four seconds on load. Scrolling stutters at 8 fps. Memory balloons to 800 MB. A ticket lands in your inbox with the subject line: "Transactions page is unusable."

Here's the thing — you didn't write bad code. You ran into a fundamental limitation of how browsers handle the DOM. They were simply never designed to hold 50,000 live nodes in memory simultaneously. Each node costs memory, each style recalculation touches the whole tree, and every scroll event triggers a repaint across thousands of elements.

Your innocent .map() looked like this:

function TransactionList({ transactions }) {
  return (
    <div className="list">
      {transactions.map(tx => (
        <TransactionRow key={tx.id} data={tx} />
      ))}
    </div>
  );
}

It works beautifully with 50 rows in development. With 47,000, the browser quietly has a breakdown. This isn't an edge case — it's the default failure mode of list rendering at scale, and nearly every frontend developer encounters it eventually.

The fix has an elegant name: list virtualization.

Before We Go There — The Simpler Approaches

It's worth pausing here, because virtualization isn't always the right answer. There are simpler options, and being honest about their trade-offs helps you reach for the right tool.

No optimization gives you a perfect UX — seamless scroll, no page breaks — but at the cost of a huge DOM, very high memory usage, and scroll performance that falls apart at scale.

Pagination keeps the DOM small and memory low, and is trivial to implement. The catch is that users lose scroll position and spatial context every time they flip a page. Fine for search results; frustrating for a data grid someone navigates fluidly.

Infinite scroll without virtualization feels smarter — it appends items as you scroll, so there's no jarring page flip. But it never removes old items. Memory grows without bound, and what starts fast degrades the longer someone scrolls. It's the "feels good at 200 rows, breaks at 2,000" approach.

CSS content-visibility: auto is a newer, genuinely underrated option. It's a single CSS property that tells the browser to skip layout and painting for off-screen elements. Completely free. It can meaningfully improve scroll performance — but it doesn't reduce the number of DOM nodes, so memory and initial parse time still scale with your list length.

None of these maintain a small DOM and a smooth, infinite scroll experience at the same time. That's the specific gap list virtualization fills — a constant ~20 DOM nodes, constant low memory, 60 fps scrolling, and seamless UX all at once.

The Surprisingly Simple Idea Behind It

When you first understand how list virtualization works, you might feel a little cheated by how obvious it is. The core idea is this: only render the rows the user can actually see, plus a small buffer above and below. As they scroll, you recycle the existing DOM nodes — swapping their content to reflect whatever has scrolled into view — rather than creating new ones.

Think of it like a stage with a small window. The audience can only see what's in the window, so you only need to dress the set that's currently visible. As the camera pans, you swap out the scenery just fast enough that no one notices.

In practice, a virtualizer does three things to make this illusion work:

  • A tall outer spacer div is given a height equal to totalItems × rowHeight. This tricks the browser's scrollbar into thinking the full list exists — it shows the correct proportional position and length.
  • A small set of absolutely-positioned rows (~18–22 nodes, regardless of list length) are placed at the exact pixel offset corresponding to whatever items are currently visible.
  • As the user scrolls, rows that drift out of the buffer zone are recycled — their content is updated to reflect new items without destroying the underlying DOM node.

The result: the browser believes it's rendering a massive list. Your actual DOM stays tiny.

Fixed Heights vs. Variable Heights — Choosing the Right Path

There are two main variants of the virtualization algorithm, and picking the wrong one is a common source of subtle bugs.

The Fast Path: Fixed Height

If every row in your list is the same height, the math is beautifully simple:

const firstVisible = Math.floor(scrollTop / rowHeight);
const lastVisible  = Math.min(
  totalItems - 1,
  Math.ceil((scrollTop + viewportHeight) / rowHeight)
);

The library calculates the visible range in O(1) — no scanning, no measuring. This is the fast path. Whenever you control your row content, try to keep heights uniform.

The Necessary Complexity: Variable Height

For real-world content — chat messages of different lengths, expandable accordions, cards with wrapping text — rows will have different heights. The virtualizer must either measure each rendered row and cache its height, or accept estimated heights and correct them after measurement. Position calculations become O(n) or require a binary search over a cached offset array.

It's more complex, but it's the only way to handle genuinely dynamic content.

What Happens Every Frame

During scrolling, the virtualizer runs this cycle on every animation frame — typically within 16 milliseconds, fast enough to be completely imperceptible:

  1. A scroll event fires — the container's scrollTop is read
  2. The visible range is calculated — firstIndex and lastIndex are derived from scroll position and row heights
  3. Rows are rendered and positioned — only the visible items, plus a buffer, are placed at their correct absolute pixel offsets
  4. Off-screen nodes are recycled — any rows that left the buffer zone are unmounted or have their content swapped

This cycle repeats on every scroll tick, doing a tremendous amount of quiet work so the user experiences something that feels effortless.

How the Libraries Stack Up

These numbers are based on a benchmark of 10,000 rows of simple text content, run in Chrome 122 on a MacBook M2. Real-world results will vary by row complexity and device. For independent comparisons, see LogRocket's react-virtualized vs react-window analysis and this TanStack Virtual vs React-Virtualized deep dive.

Without any virtualization, rendering 10,000 rows takes around 2,400 ms, consumes ~680 MB of memory, produces about 9 fps scroll performance, and creates 10,000 DOM nodes.

Here's how each library compares:

  • react-window — ~38 ms render, ~28 MB memory, 60 fps scroll, ~18 DOM nodes, 2.2 KB gzipped
  • @tanstack/virtual — ~42 ms render, ~31 MB memory, 58 fps scroll, ~20 DOM nodes, 4.1 KB gzipped
  • react-virtuoso — ~55 ms render, ~35 MB memory, 57 fps scroll, ~22 DOM nodes, 14 KB gzipped
  • react-virtualized — ~48 ms render, ~33 MB memory, 58 fps scroll, ~20 DOM nodes, 25 KB gzipped
  • vue-virtual-scroller — ~45 ms render, ~30 MB memory, 58 fps scroll, ~20 DOM nodes, 6 KB gzipped
  • @angular/cdk — ~50 ms render, ~32 MB memory, 57 fps scroll, ~20 DOM nodes, part of CDK bundle

The most important thing to notice isn't the differences between libraries — they're remarkably close to each other on every dimension. The real story is the contrast between any virtualized library and doing nothing at all.

The improvement in plain numbers

  • Render time: 63× faster (2,400 ms → 38 ms)
  • Memory usage: 24× less (680 MB → 28 MB)
  • Scroll performance: 6.7× smoother (9 fps → 60 fps)
  • DOM nodes: 99.8% fewer (10,000 → ~20)

Feature comparison by library

react-window — React only. Supports fixed and variable height, grid layout, and horizontal scrolling. Infinite loading requires a plugin (react-window-infinite-loader). Headless: no. Actively maintained.

@tanstack/virtual — Framework-agnostic (React, Vue, Solid, etc.). Supports fixed and variable height, auto-measurement, grid layout, horizontal scrolling, built-in infinite loading, and is fully headless. Actively maintained.

react-virtuoso — React only. Supports fixed and variable height with automatic measurement, partial grid support, built-in infinite loading. No horizontal scroll, not headless. Actively maintained.

react-virtualized — React only. Full-featured: fixed and variable height, CellMeasurer for auto-measurement, grid support, built-in infinite loading, horizontal scrolling. Minimal maintenance — the author recommends react-window as its leaner successor.

vue-virtual-scroller — Vue only. Fixed and variable height, no grid, no built-in infinite loading (manual). Actively maintained.

@angular/cdk Scrolling module — Angular only. Fixed and variable height, no built-in infinite loading (manual). Ships as part of the CDK, not a standalone package. Actively maintained.

Code Examples

react-window — Fixed Height List

The minimal setup. If you know your row heights, this is all you need:

import { FixedSizeList as List } from 'react-window';

const Row = ({ index, style }) => (
  <div style={style}>
    Transaction #{index} — ${(Math.random() * 1000).toFixed(2)}
  </div>
);

export function TransactionList({ items }) {
  return (
    <List
      height={600}          // viewport height in px
      itemCount={items.length}
      itemSize={52}         // row height in px — must be fixed
      width="100%"
    >
      {Row}
    </List>
  );
}

One thing catches almost everyone the first time: the style prop injected into each Row carries the absolute positioning. You must spread it onto your row's root element. Skip it, and every row stacks at top: 0.

For variable height rows, swap to VariableSizeList:

import { VariableSizeList as List } from 'react-window';

const getItemSize = index => itemHeights[index]; // your cached heights

<List
  height={600}
  itemCount={items.length}
  itemSize={getItemSize}
  width="100%"
>
  {Row}
</List>

@tanstack/virtual — The Headless Hook

TanStack Virtual gives you the virtualization math without any markup — ideal when your design system already owns the DOM structure, or when you need framework flexibility:

import { useVirtualizer } from '@tanstack/react-virtual';
import { useRef } from 'react';

export function TransactionList({ items }) {
  const parentRef = useRef(null);

  const rowVirtualizer = useVirtualizer({
    count: items.length,
    getScrollElement: () => parentRef.current,
    estimateSize: () => 52,
    overscan: 5,           // buffer rows above and below viewport
  });

  return (
    <div ref={parentRef} style={{ height: 600, overflow: 'auto' }}>
      {/* This div's height tricks the scrollbar */}
      <div style={{ height: rowVirtualizer.getTotalSize(), position: 'relative' }}>
        {rowVirtualizer.getVirtualItems().map(virtualRow => (
          <div
            key={virtualRow.index}
            style={{
              position: 'absolute',
              top: 0,
              left: 0,
              width: '100%',
              transform: `translateY(${virtualRow.start}px)`,
            }}
          >
            {items[virtualRow.index].description}
          </div>
        ))}
      </div>
    </div>
  );
}

getTotalSize() returns the spacer height. getVirtualItems() returns only the rows currently in the render window. Everything else is your code — no imposed classnames, no wrapper components, total styling freedom.

react-virtuoso — Auto-Measuring Heights

Virtuoso's standout feature is that it measures row heights automatically after render, so you don't need to maintain a height cache at all:

import { Virtuoso } from 'react-virtuoso';

export function TransactionList({ items }) {
  return (
    <Virtuoso
      style={{ height: 600 }}
      totalCount={items.length}
      itemContent={index => (
        <div style={{ padding: '12px 16px', borderBottom: '1px solid #eee' }}>
          <strong>{items[index].title}</strong>
          <p>{items[index].description}</p>
        </div>
      )}
    />
  );
}

Virtuoso renders each item, measures it, and adjusts offsets accordingly. It's especially well-suited to chat UIs, social feeds, or any list where content height is determined by text wrapping that you can't predict ahead of time.

The Pitfalls Worth Knowing About

Virtualization adds a small layer of complexity, and there are a handful of mistakes that come up so often they're almost a rite of passage.

Defining row components inside the parent

// Bad — MyRow is redefined on every parent render
<List itemContent={index => <MyRow item={items[index]} />} />

// Good — defined outside, stable reference
const MyRow = ({ item }) => <div>{item.title}</div>;
<List itemContent={index => <MyRow item={items[index]} />} />

When you define a component inline, React treats it as a new component type on every parent render and forces every visible row to unmount and remount. Define row components outside the parent, or memoize them.

Forgetting to spread the style prop

// Bad — all rows stack at top: 0
const Row = ({ index, style }) => (
  <div>Row {index}</div>  // ← style not applied!
);

// Good
const Row = ({ index, style }) => (
  <div style={style}>Row {index}</div>
);

Unstable keys with dynamic data

// Bad — index as key causes unnecessary unmounts on filter/sort
items.map((item, index) => <Row key={index} />)

// Good — stable ID
items.map(item => <Row key={item.id} />)

List inside a flex column without explicit height

// Bad — flex child has no concrete height; the virtualizer can't calculate what's visible
<div style={{ display: 'flex', flexDirection: 'column' }}>
  <Header />
  <VirtualList style={{ flex: 1 }} />  {/* ← no height! */}
</div>

// Good — explicit height
<div style={{ display: 'flex', flexDirection: 'column', height: '100vh' }}>
  <Header style={{ height: 60 }} />
  <VirtualList style={{ flex: 1, height: 'calc(100vh - 60px)' }} />
</div>

Measuring before fonts load

For variable height lists, the measurement pass can run before your web fonts finish loading. Measuring text in the fallback font gives wrong heights, causing layout jumps when the real font arrives.

// Defer render until fonts are ready
await document.fonts.ready;
setReady(true);

Advanced Patterns

Windowed Infinite Loading

This pattern combines virtualization with paginated data fetching: you only request the next page when the user's scroll position approaches the bottom of already-loaded data. react-window-infinite-loader integrates this with react-window directly — when the virtualizer asks for row index 500 and you've only fetched 400, it triggers a fetch transparently.

import { FixedSizeList as List } from 'react-window';
import InfiniteLoader from 'react-window-infinite-loader';

function InfiniteList({ hasNextPage, isNextPageLoading, items, loadNextPage }) {
  const itemCount = hasNextPage ? items.length + 1 : items.length;
  const isItemLoaded = index => !hasNextPage || index < items.length;

  return (
    <InfiniteLoader
      isItemLoaded={isItemLoaded}
      itemCount={itemCount}
      loadMoreItems={isNextPageLoading ? () => {} : loadNextPage}
    >
      {({ onItemsRendered, ref }) => (
        <List
          ref={ref}
          height={600}
          itemCount={itemCount}
          itemSize={52}
          onItemsRendered={onItemsRendered}
          width="100%"
        >
          {({ index, style }) => (
            <div style={style}>
              {isItemLoaded(index) ? items[index].title : 'Loading...'}
            </div>
          )}
        </List>
      )}
    </InfiniteLoader>
  );
}

Bidirectional Virtualization (Chat UIs)

Chat apps have a unique challenge: new messages arrive at the bottom, but users can scroll arbitrarily far back into history. Prepending old messages without jumping the scroll position is notoriously tricky. react-virtuoso handles this natively with its firstItemIndex prop, which decrements as history loads without disturbing where the user is currently reading.

<Virtuoso
  firstItemIndex={firstItemIndex}   // decrements as history loads
  initialTopMostItemIndex={messages.length - 1}
  data={messages}
  startReached={loadOlderMessages}  // triggered when user hits the top
  itemContent={(index, message) => <Message data={message} />}
/>

Virtualized Grids

The same idea extends to two dimensions — only render the rows and columns currently in the viewport:

import { FixedSizeGrid as Grid } from 'react-window';

<Grid
  columnCount={100}
  columnWidth={120}
  height={600}
  rowCount={10000}
  rowHeight={52}
  width={800}
>
  {({ columnIndex, rowIndex, style }) => (
    <div style={style}>
      {rowIndex},{columnIndex}
    </div>
  )}
</Grid>

Dynamic Data with Memoization

If each row renders a complex component, wrap it in React.memo to prevent re-renders when unrelated state changes upstream:

const MemoizedRow = React.memo(({ item, style }) => (
  <div style={style}>
    <Avatar src={item.avatar} />
    <span>{item.name}</span>
    <span>{item.amount}</span>
  </div>
));

Pair this with useCallback for any handlers passed as props — otherwise every parent render creates new function references and invalidates every row's memo check.

When Should You Actually Virtualize?

A simple decision path to save you the guesswork:

  1. Is the list user-scrollable? If not — no virtualization needed.
  2. Does it have fewer than ~300 items in real usage? If yes — a simple render or content-visibility is enough.
  3. Are row heights uniform? If no — reach for react-virtuoso, which auto-measures heights.
  4. Do you need full styling control or framework flexibility? If yes — @tanstack/virtual is the headless hook for you.
  5. Otherwise — react-window with FixedSizeList is your default.

Which Library Belongs in Your Project?

react-window is the sensible default for most React projects. It's lightweight at ~2 KB gzipped, actively maintained, and has a clean separation between FixedSizeList and VariableSizeList. When in doubt, start here.

@tanstack/virtual earns its place when you need framework flexibility, are already using TanStack Table, or want full control over rendering. It ships no UI — just the virtualization logic as a hook — which means zero imposed styling and clean integration with any design system.

react-virtuoso has one compelling advantage: it measures rows automatically after render. If your rows have truly unpredictable heights — chat messages, rich content, anything that wraps dynamically — Virtuoso removes a lot of manual bookkeeping. It's heavier at ~14 KB, but that weight often pays for itself.

vue-virtual-scroller is the natural choice for Vue apps — well-maintained within the Vue ecosystem, supporting both fixed and dynamic sizing without friction.

react-virtualized built the vocabulary that all of these libraries speak. It's a remarkable piece of engineering, and you'll find it in countless production codebases. For new projects, though, its author has explicitly directed people toward react-window as the leaner, more focused successor — and the 25 KB bundle overhead is hard to justify when the alternatives are this good.

A Technique Worth Having in Your Toolkit

List virtualization is one of those rare performance techniques that delivers dramatic, measurable wins without demanding a full architectural rethink. The core idea — render only what's visible, recycle what isn't — is simple enough to implement from scratch, yet modern libraries mean you almost never need to.

The numbers are worth sitting with for a moment. For a 10,000-row list, virtualization delivers 63× faster initial renders, 24× lower memory usage, and 6.7× smoother scrolling compared to a raw .map() render. The DOM node count holds at a constant ~20 whether your list has 1,000 items or 1,000,000.

The question of whether to virtualize is easy: if your list can exceed a few hundred items in real usage, open the Browser DevTools Performance tab and have a look. If list rendering shows up anywhere near the top, virtualization is almost certainly worth the one-afternoon investment — and the user on the other end of that frozen transactions page will thank you for it.

Benchmarks: 10,000 rows of simple text content, Chrome 122, MacBook M2. Real-world results vary by row complexity, device capability, and network conditions.
